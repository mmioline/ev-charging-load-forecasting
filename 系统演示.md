
# EV 充电桩实时监测系统演示方案

## 端口说明

```
18010  用户服务
18011  站点服务
18012  充电记录服务
18013  预测服务
18080  前端页面
```

MySQL / Redis 仅在 Docker 内网使用，不对外暴露。

---

## 一、启动确认

```bash
docker compose ps
curl -sS http://localhost:18010/health | python3 -m json.tool
```

如果服务不干净，重置：

```bash
docker compose down -v --remove-orphans
docker compose up -d --build --force-recreate
```

> WSL 重启后如果端口不通，运行：
> `sudo sysctl -w net.ipv4.conf.all.route_localnet=1`
> 然后 `docker compose up -d`

## 四、创建三个充电站

```bash
A_ID=$(curl -sS -X POST http://localhost:18011/stations \
  -H "Authorization: Bearer $ALICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"科技园A站","location":"深南大道1号","capacity":60.0,"slots":6}' \
  | python3 -c 'import sys,json; print(json.load(sys.stdin)["id"])')

B_ID=$(curl -sS -X POST http://localhost:18011/stations \
  -H "Authorization: Bearer $ALICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"总部基地B站","location":"科苑路88号","capacity":160.0,"slots":16}' \
  | python3 -c 'import sys,json; print(json.load(sys.stdin)["id"])')

C_ID=$(curl -sS -X POST http://localhost:18011/stations \
  -H "Authorization: Bearer $ALICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"老城区C站","location":"人民南路99号","capacity":100.0,"slots":10}' \
  | python3 -c 'import sys,json; print(json.load(sys.stdin)["id"])')

echo "A=$A_ID  B=$B_ID  C=$C_ID"
```

确认：

```bash
curl -sS http://localhost:18011/stations \
  -H "Authorization: Bearer $ALICE_TOKEN" | python3 -m json.tool
```

## 五、模拟 C 站超时异常

```bash
# 将 C 站设为充电中
curl -sS -X PUT http://localhost:18011/stations/$C_ID/status \
  -H "Authorization: Bearer $ALICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"status":"充电中"}' | python3 -m json.tool

# 将 last_updated_at 回拨 25 小时
docker exec ev_charging_db mysql -uroot -proot ev_charging \
  -e "UPDATE stations SET last_updated_at = DATE_SUB(NOW(), INTERVAL 25 HOUR) WHERE id = $C_ID;"

# 验证
docker exec ev_charging_db mysql -uroot -proot ev_charging \
  -e "SELECT id, name, status, last_updated_at FROM stations WHERE id = $C_ID;"
```

## 六、注入历史数据并制造空闲概率差异

### 6.1 注入 100 条充电记录

```bash
source venv/bin/activate
SEED_STATION_IDS="$A_ID,$B_ID" python services/seed_data.py
```

### 6.2 时间分散（B 站保持分散，空闲率高）

```bash
docker exec ev_charging_db mysql -uroot -proot ev_charging \
  -e "SET @row := 0;
      UPDATE charging_records
      SET created_at = DATE_SUB(NOW(), INTERVAL ((@row := @row + 1) % 30) DAY)
      WHERE station_id = $B_ID;"
```

### 6.3 A 站记录集中到当前小时（空闲率低）

将 A 站大量记录集中在当前小时，duration 设为 90 分钟，
使该时段并发数远超 6 桩，空闲概率大幅下降：

```bash
docker exec ev_charging_db mysql -uroot -proot ev_charging \
  -e "UPDATE charging_records
      SET created_at = DATE_FORMAT(NOW(), '%Y-%m-%d %H:00:00'),
          duration_minutes = 90
      WHERE station_id = $A_ID;"
```

### 6.4 验证记录分布

```bash
docker exec ev_charging_db mysql -uroot -proot ev_charging \
  -e "SELECT station_id, COUNT(*) AS cnt FROM charging_records GROUP BY station_id;"
```

## 八、正式演示

打开前端页面：`http://localhost:18080`

点击 **Alice** 快捷按钮填入账号，登录。

### 8.1 查看初始状态

前端点击「全部站点」，观察：
- A 站：空闲，6桩
- B 站：空闲，16桩
- C 站：充电中（即将被判定为异常）

### 8.2 C 站超时自动判定为异常

```bash
curl -sS http://localhost:18011/stations/status/available \
  -H "Authorization: Bearer $ALICE_TOKEN" | python3 -m json.tool
```

预期：返回列表只含 A、B 两站，C 站已被系统标记为「异常」。

前端点「只看空闲」，C 站消失。

验证 Redis 缓存同步：

```bash
docker exec ev_station_redis redis-cli GET "station:status:$C_ID"
```

预期：`"status": "异常"`

### 8.3 Alice 向 A 站发起充电

```bash
ALICE_RECORD_ID=$(curl -sS -X POST http://localhost:18012/charging \
  -H "Authorization: Bearer $ALICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"station_id\": $A_ID, \"duration_minutes\": 45, \"kwh_consumed\": 22.5}" \
  | python3 -c 'import sys,json; print(json.load(sys.stdin)["id"])')

echo "Alice 充电记录 ID: $ALICE_RECORD_ID"
```

验证 A 站状态变为「充电中」：

```bash
curl -sS http://localhost:18011/stations/$A_ID/status \
  -H "Authorization: Bearer $ALICE_TOKEN" | python3 -m json.tool

docker exec ev_station_redis redis-cli GET "station:status:$A_ID"
```

### 8.4 Bob 抢占 A 站被拒绝（充电冲突演示）

```bash
curl -sS -X POST http://localhost:18012/charging \
  -H "Authorization: Bearer $BOB_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"station_id\": $A_ID, \"duration_minutes\": 30, \"kwh_consumed\": 15.0}"
```

预期返回：

```json
{"detail": "站点当前状态为「充电中」，无法发起充电"}
```

前端切换到 Bob 账号，选 A 站点击「发起充电」，actionMsg 区域显示红色报错。

### 8.5 Bob 转向 B 站充电成功

```bash
curl -sS -X POST http://localhost:18012/charging \
  -H "Authorization: Bearer $BOB_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"station_id\": $B_ID, \"duration_minutes\": 30, \"kwh_consumed\": 15.0}" \
  | python3 -m json.tool
```

### 8.6 订单数据隔离

Alice 查自己的记录，正常返回：

```bash
curl -sS http://localhost:18012/charging/$ALICE_RECORD_ID \
  -H "Authorization: Bearer $ALICE_TOKEN" | python3 -m json.tool
```

Bob 查 Alice 的记录，被拒绝（404，不泄露记录存在性）：

```bash
curl -i -sS http://localhost:18012/charging/$ALICE_RECORD_ID \
  -H "Authorization: Bearer $BOB_TOKEN"
```

### 8.7 鉴权拦截

无 Token：

```bash
curl -i -sS http://localhost:18011/stations
```

伪造 Token：

```bash
curl -i -sS http://localhost:18011/stations \
  -H "Authorization: Bearer this.is.a.fake.token"
```

两者均返回 401。

### 8.8 负荷预测对比（A/B 站空闲概率差异）

```bash
# A 站：记录集中，预期空闲概率低
curl -sS http://localhost:18013/predict/$A_ID | python3 -m json.tool

# B 站：记录分散，预期空闲概率高
curl -sS http://localhost:18013/predict/$B_ID | python3 -m json.tool
```

预期：A 站 `idle_probability_next_hour` 明显低于 B 站，体现系统对不同负荷模式的区分能力。

## 演示路径总览

```
启动确认
  │
  ├─ 准备阶段（演示前执行）
  │   ├─ 注册 alice / bob / pytest_admin / manual_tester
  │   ├─ 创建 A站(6桩) / B站(16桩) / C站(10桩)
  │   ├─ C站回拨时间 → 触发超时异常
  │   ├─ 注入100条记录
  │   ├─ A站记录集中当前小时 → 空闲概率低
  │   ├─ B站记录时间分散 → 空闲概率高
  │   └─ 训练LSTM模型
  │
  └─ 演示阶段（现场执行）
      ├─ [初始状态] 三站全览
      ├─ [异常判定] 查询空闲站点 → C站自动标记异常 + Redis验证
      ├─ [充电①] Alice → A站成功 → 状态变「充电中」
      ├─ [冲突拦截] Bob → A站被拒（400）
      ├─ [充电②] Bob → B站成功
      ├─ [数据隔离] Bob查Alice订单 → 404
      ├─ [鉴权] 无Token/伪Token → 401
      ├─ [预测对比] A站低空闲率 vs B站高空闲率
      └─ [自动化测试] pytest → HTML报告
```
